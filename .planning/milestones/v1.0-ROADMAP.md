# Milestone v1.0: Dual Camera Model Rollout

**Status:** ✅ SHIPPED 2026-04-22
**Phases:** 1-5
**Total Plans:** 15

## Overview

This milestone upgraded the shared-scene renderer so `pinhole32` and `equi62_lut1d` are first-class per-camera models across scene definitions, offline CPU rendering, realtime GPU/viewer rendering, and default camera construction. It also switched implicit defaults to fisheye, added a deterministic intrinsics derivation utility, and closed on direct reference-backed regression coverage.

## Phases

### Phase 1: Shared Camera Schema

**Goal**: Scene definitions and shared camera data can represent either supported camera model per camera, along with the v1 common intrinsics needed by both render backends.  
**Depends on**: Nothing (first phase)  
**Plans**: 3 plans

Plans:

- [x] 01-01: Define the canonical `CameraSpec` contract and rig-packing adapter boundary.
- [x] 01-02: Move builtin scene presets and shared-scene consumers onto `CameraSpec`.
- [x] 01-03: Migrate YAML camera declarations and structural regressions onto the canonical schema.

**Details:**
- Requirements: `CAM-01`, `CAM-05`, `DEF-01`
- Outcome: shared `CameraSpec` and `CameraRig` contract established; builtin and YAML scene cameras migrated to explicit `model/width/height/fx/fy/cx/cy/T_bc`; legacy YAML camera fields rejected.

### Phase 2: Offline CPU Camera Models

**Goal**: Offline CPU rendering uses the selected per-camera model from shared scene data instead of falling back to pinhole-only ray generation.  
**Depends on**: Phase 1  
**Plans**: 3 plans

Plans:

- [x] 02-01: Introduce the model-aware offline primary-ray seam without forking the CPU tracer.
- [x] 02-02: Route shared-scene offline entrypoints through canonical camera data and prove preset-path model switching.
- [x] 02-03: Align viewer CPU-reference regressions with the new offline camera-model contract.

**Details:**
- Requirements: `CAM-02`
- Outcome: offline CPU rendering now honors canonical `pinhole32` and `equi62_lut1d` camera math through one shared-scene contract, with pinhole-only depth of field preserved.

### Phase 3: Realtime GPU And Viewer Camera Models

**Goal**: Realtime rendering and the viewer screen preserve each camera's selected model from shared scene data through rig packing, OptiX launches, and active viewer cameras.  
**Depends on**: Phase 1  
**Plans**: 3 plans

Plans:

- [x] 03-01: Replace the viewer's hardcoded pinhole rig with a spec-driven builder aligned to authored camera specs.
- [x] 03-02: Preserve packed camera model data through OptiX upload and recover equi realtime rendering regressions.
- [x] 03-03: Add file-backed and mixed-model end-to-end regressions for realtime factory, viewer reload, and four-camera rigs.

**Details:**
- Requirements: `CAM-03`, `CAM-04`
- Outcome: realtime factory, viewer rigging, active OptiX camera selection, reload/rescan flows, and mixed-model rigs preserve per-camera model and intrinsics end to end.

### Phase 4: Default Intrinsics And Fisheye Defaults

**Goal**: New and default camera setups derive deterministic v1 intrinsics from image resolution plus horizontal FOV, with fisheye becoming the project default while pinhole stays explicitly available.  
**Depends on**: Phase 2, Phase 3  
**Plans**: 3 plans

Plans:

- [x] 04-01: Add the shared default-intrinsics derivation helper and thin utility wrapper.
- [x] 04-02: Move builtin default CPU/realtime camera helpers onto the fisheye-first shared derivation path.
- [x] 04-03: Switch the viewer no-arg default rig to fisheye and lock remaining default-entry regressions.

**Details:**
- Requirements: `DEF-02`, `DEF-03`, `DEF-04`, `DEF-05`, `DEF-06`
- Outcome: implicit/default camera construction is now fisheye-first, explicit authored pinhole presets remain pinhole, and default intrinsics are derived from `resolution + hfov`.

### Phase 5: Camera Model Regression Coverage

**Goal**: Automated coverage proves both camera models match the reference math and remain consistent across offline and realtime rendering after fisheye becomes the default.  
**Depends on**: Phase 4  
**Plans**: 3 plans

Plans:

- [x] 05-01: Anchor representative `pinhole32` and `equi62_lut1d` math directly to the bundled reference headers.
- [x] 05-02: Prove offline CPU and realtime GPU honor the same packed-camera ray contract and default-derived fisheye smoke path.
- [x] 05-03: Tighten authored-versus-default regressions and lock the widened final camera-model slice green.

**Details:**
- Requirements: `VER-01`, `VER-02`, `VER-03`, `VER-04`
- Outcome: regression coverage now directly anchors to bundled reference camera headers and proves cross-path contract consistency after the fisheye default switch.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Use `docs/reference/src-cam/cam_pinhole32.h` and `cam_equi62_lut1d.h` as the camera-model math references.
- Treat `pinhole32` and `equi62_lut1d` as first-class per-camera types across shared scene, offline, realtime, and viewer paths.
- Make fisheye the default camera model while preserving explicit pinhole support everywhere.
- Derive v1 default `fx/fy/cx/cy` from `resolution + hfov` before introducing richer calibration surfaces.
- Defer dynamic resolution, explicit calibration parameters, distortion tuning, and configurable extrinsics to the next milestone.

**Issues Resolved:**

- Eliminated pinhole-only schema assumptions in shared scene and YAML camera declarations.
- Removed offline shared-scene pinhole-only ray generation.
- Removed viewer/realtime default-rig pinhole assumptions.
- Recovered live equi realtime regression coverage and aligned CPU/GPU direction contracts.

**Issues Deferred:**

- Dynamic render-preset-controlled output resolution.
- Explicit per-camera intrinsics.
- Model-specific distortion coefficients.
- Configurable body-relative camera extrinsics through scene or preset data.

**Technical Debt Incurred:**

- Formal per-phase `*-VERIFICATION.md` artifacts were not used; this milestone relied on `*-UAT.md` as the authoritative verification surface.
