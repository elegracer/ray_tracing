# v2.0 Realtime Ray Tracing Modernization Research

**Date:** 2026-07-18
**Target:** Linux, C++23/CUDA, OptiX 9.1, RTX 3090
**Evidence baseline:** `.planning/v2.0-BASELINE.json`

## Decision Summary

The renderer should evolve as a composed-scene compiler and GPU path tracer, not as a larger collection of bespoke YAML variants. OpenUSD should own authoring, composition, hierarchy, identity, units, transforms, instancing, cameras, geometry, lights, and material binding. OpenPBR Surface, represented through MaterialX and `UsdShade`, should own the surface-material contract. The renderer should compile those standards into a compact backend IR shared by CPU reference and OptiX execution.

The first performance investment is not ReSTIR. It is a truthful profiler, GPU-resident frame data, and a real OptiX temporal AOV denoiser. The current `realtime` profile is slower than the `quality` profile because `OptixDenoiserWrapper` is a CPU clamp loop. Fixing that path improves both quality and latency while creating the motion/AOV infrastructure later required by ReSTIR and NRD.

## Repository Findings

### What is already useful

- A backend-neutral `scene::SceneIR` feeds both CPU and realtime adapters.
- OptiX rendering already provides beauty, albedo, normal, depth, camera history, accumulation, per-camera CUDA streams, and acceleration structures.
- The project has hot-reloadable YAML scenes, OBJ import, two camera models, four-camera rendering, benchmark JSON/CSV, and 54 passing tests.
- Scene preparation is already separated from repeated frame rendering, which is the right lifecycle for persistent GPU resources.

### Correctness gaps

- `SceneIR` has integer-indexed flat arrays and no stable prim path, hierarchy, scale, mesh normals, UVs, tangents, interpolation modes, geometry subsets, light schemas, or prototype instances.
- Materials are a five-way variant (`diffuse`, `metal`, `dielectric`, `emissive`, `isotropic`) rather than an energy-conserving layered BSDF.
- GPU diffuse sampling uses `normal + random_unit_vector`; metal uses fuzzed reflection; dielectric uses a simple Schlick decision. There is no matched BSDF evaluation/PDF contract, microfacet distribution, visible-normal sampling, transmission roughness, anisotropy, coat, fuzz, thin film, or subsurface model.
- Direct lighting linearly scans every sphere, quad, and triangle at every hit. Quad and triangle lights are sampled at their centroids without an area PDF or emitter cosine term. This is neither scalable nor an unbiased MIS estimator.
- Image-texture failures and startup scene-catalog failures can be silent, so asset correctness is not fail-closed.

### Performance and measurement gaps

- `OptixDenoiserWrapper::denoise_in_place` ignores albedo, normal, width, and height, then clamps every beauty value on the CPU.
- Four cameras and denoisers run concurrently, but reports sum their work times and subtract that sum from wall time. Negative `host_overhead_ms` is therefore expected and invalid.
- Every frame downloads four full beauty/albedo/normal/depth buffers to the host before display or denoise. The viewer does not use CUDA/OpenGL interop.
- `RendererPool` creates fresh `std::async` tasks each frame, launch parameters are allocated per launch, and acceleration preparation always enters a rebuild path despite the method name `build_or_refit_accels`.
- The benchmark has no warmup, device metadata, GPU-memory metric, confidence interval, or critical-path stage definition.

## Measured Baseline

The following steady-state figures exclude frame zero. All runs use `smoke`, four 640x480 cameras, 100 requested frames, no PNG writes, RTX 3090, CUDA 13.0, and OptiX 9.1.

| Profile | SPP / bounces | Current denoise | Avg frame | P95 frame | FPS |
|---|---:|---|---:|---:|---:|
| `realtime` | 1 / 2 | CPU clamp stub | 42.85 ms | 45.79 ms | 23.34 |
| `balanced` | 2 / 4 | CPU clamp stub | 43.35 ms | 46.78 ms | 23.07 |
| `quality` | 4 / 8 | off | 28.96 ms | 32.69 ms | 34.53 |

The quality inversion is decisive evidence that the current post-process and host transfer path dominate this workload. Stage sums in the raw report are work totals, not latency totals, and must not be used for speedup claims until the profiler contract is corrected.

## Public Technique Assessment

| Technique | Public primary evidence | Repository fit | Priority |
|---|---|---|---|
| OptiX temporal AOV denoiser | [OptiX 9 denoiser guide](https://raytracing-docs.nvidia.com/optix9/guide/index.html#nvidia-ai-denoiser) and vendored 9.1 headers expose temporal AOV, flow, previous output, albedo, and normal guides | Native to the current API and lowest integration risk; keep beauty and guides on GPU | P0 |
| ReSTIR DI/GI/PT | [RTXDI 3.0](https://github.com/NVIDIA-RTX/RTXDI) includes DI, GI, and PT implementations; [GRIS/ReSTIR PT paper](https://research.nvidia.com/publication/2022-07_generalized-resampled-importance-sampling-foundations-restir) defines the estimator | Algorithms are relevant, but RTXDI is an HLSL/NVRHI reference rather than a drop-in CUDA library. Port DI only after correct light sampling and PDFs | P1 DI, P2 GI/PT |
| NRD RELAX/REBLUR | [NVIDIA NRD](https://github.com/NVIDIA-RTX/NRD) is a public spatiotemporal denoiser using normal, roughness, view-Z, and motion guides | Strong comparison target, but its shipped integration is D3D/Vulkan shader oriented. Evaluate after GPU interop; do not replace the native OptiX P0 path blindly | P2 experiment |
| Neural radiance caching | [Original NVIDIA publication](https://research.nvidia.com/publication/2021-06_real-time-neural-radiance-caching-path-tracing) trains a cache online for dynamic GI | Promising for many-bounce GI, but materially raises training, stability, and validation complexity | P2 research |
| Shader execution reordering | [NVIDIA SER overview](https://developer.nvidia.com/blog/improve-shader-performance-and-in-game-frame-rates-with-shader-execution-reordering/) describes architecture-specific divergence reduction | Must remain capability-gated and cannot be an RTX 3090 acceptance dependency | P2 hardware path |

ReSTIR and denoising are complementary: ReSTIR allocates scarce path samples better; a temporal denoiser reconstructs residual noise. Neither compensates for an incorrect BSDF/PDF pair or missing motion/disocclusion data.

## OpenUSD Alignment Contract

Use the OpenUSD SDK to compose a `UsdStage`; do not reimplement USD composition in project code. Compile a selected time sample of the composed stage into `SceneIR v2`.

The canonical contract must preserve:

- stage metadata: `metersPerUnit`, up axis, time codes, default prim, asset resolver context;
- stable prim identity using `SdfPath`, hierarchy, visibility, purpose, kind, and instance/prototype relationships;
- complete affine transforms, reset-xform-stack semantics, and motion samples;
- `UsdGeomMesh` topology, points, normals, tangents, primvars, interpolation, indices, orientation, subdivision scheme, and `UsdGeomSubset` material partitions;
- `UsdGeomCamera` and the existing project camera extension without changing USD camera conventions silently;
- `UsdLux` distant, dome, sphere, disk, rect, and cylinder lights with intensity, exposure, color temperature, normalize, shaping, and texture semantics;
- `UsdShade` material binding resolution and connected shader networks;
- explicit diagnostics for unsupported active schemas instead of silent data loss.

Primary references: [UsdGeom](https://openusd.org/release/api/usd_geom_page_front.html), [UsdShade](https://openusd.org/release/api/usd_shade_page_front.html), and [UsdLux](https://openusd.org/release/api/usd_lux_page_front.html).

The first core-contract slice was checked against the official [SdfPath](https://openusd.org/release/api/class_sdf_path.html), [UsdGeomXformable](https://openusd.org/release/api/class_usd_geom_xformable.html), [UsdGeomImageable](https://openusd.org/release/api/class_usd_geom_imageable.html), [UsdStage](https://openusd.org/release/api/class_usd_stage.html), [up-axis](https://openusd.org/release/api/group___usd_geom_up_axis__group.html), and [linear-units](https://openusd.org/release/api/group___usd_geom_linear_units__group.html) contracts. Native v2 metadata uses the OpenUSD centimeter fallback; the compatibility frontend explicitly authors the existing project's one-meter unit convention.

The camera slice was checked against the official [UsdGeomCamera](https://openusd.org/release/api/class_usd_geom_camera.html) and [GfCamera](https://openusd.org/release/api/class_gf_camera.html) contracts. OpenUSD cameras are always local Y-up, look down local -Z in a right-handed frame regardless of stage up axis, use scene units for clipping/focus distance, and tenths of a scene unit for filmback and focal length. SceneIR v2 therefore stores those standard fields directly, converts the renderer's +Z-forward/+Y-down pose explicitly, and keeps `pinhole32`/`equi62_lut1d` pixel calibration and distortion as capability-diagnosed renderer extensions rather than pretending fisheye is a standard USD projection.

Asset identity was checked against the official [SdfAssetPath](https://openusd.org/release/api/class_sdf_asset_path.html) contract. Authored, expression-evaluated, and resolver-produced paths are distinct values; only the authored asset path is serialized, and `SdfAssetPath` itself does not evaluate or resolve. SceneIR v2 mirrors those fields without implementing a competing resolver. The YAML and OBJ/MTL compatibility frontends now preserve the original authored texture token separately from the normalized execution path, while legacy builtin paths use an explicit same-path fallback until the OpenUSD resolver frontend exists.

YAML remains a compatibility frontend. It should translate into the same v2 contract and later become an optional convenience format, not the semantic source of truth.

## OpenPBR Alignment Contract

Adopt the official MaterialX node definition `ND_open_pbr_surface_surfaceshader` and preserve its names, types, defaults, and version. The current [OpenPBR repository](https://github.com/AcademySoftwareFoundation/OpenPBR) publishes the specification and a [MaterialX reference implementation](https://github.com/AcademySoftwareFoundation/OpenPBR/blob/main/reference/open_pbr_surface.mtlx). The implemented contract is pinned to the official OpenPBR `v1.1.1` release and the matching node/stdlib definitions shipped by MaterialX `v1.39.5`; future upgrades must compare the official nodedef before changing serialized defaults or validation.

The IR must represent a parameter as either a typed constant or a connected texture/node output, including UV set, transform, wrap/filter mode, and source color space. The first evaluator slice implements base weight/color/diffuse roughness/metalness, specular weight/color/roughness/IOR/anisotropy, transmission weight/color/depth, emission luminance/color, opacity, thin-walled, normal, and tangent. Coat, fuzz, thin film, dispersion, and subsurface follow without changing the authored contract.

Implementation rules:

- use matched `evaluate`, `sample`, and `pdf` functions for every lobe;
- start with GGX visible-normal sampling, generalized Schlick/dielectric Fresnel, Oren-Nayar-compatible diffuse behavior, multiple-scattering energy compensation, and Russian roulette based on path throughput;
- keep radiometric values linear and attach explicit color-space/units metadata at asset boundaries;
- treat OpenPBR volume semantics separately from OpenPBR Surface rather than hiding the current isotropic medium inside a surface variant;
- preserve unsupported non-default authored inputs and emit a validation error until their evaluator exists;
- translate legacy materials through documented compatibility mappings and image-regression gates. Approximate mappings such as `fuzz -> roughness` must never be described as lossless.

## Ordered Delivery Plan

### P0: Make measurements and reconstruction truthful

1. Define wall-clock critical path, per-camera work, GPU event time, transfer time, and overlap separately; add warmup and environment metadata.
2. Keep beauty/albedo/normal/depth/motion on the GPU and implement OptiX temporal AOV denoising with history reset/disocclusion tests.
3. Remove per-frame launch-parameter allocation and replace fresh `std::async` creation with persistent scheduling.
4. Gate P0 on improved objective image error and `realtime` P95 no worse than 45.79 ms; the stretch gate is below the current no-denoise quality P95 of 32.69 ms.

### P1: Standard contracts and scalable direct lighting

1. Introduce `SceneIR v2` with stable prim paths, units, hierarchy, affine transforms, geometry primvars/subsets, lights, cameras, instancing, and diagnostic capability flags.
2. Introduce the versioned OpenPBR parameter block and shared CPU/GPU BSDF conformance tests, then migrate legacy materials.
3. Add optional OpenUSD and MaterialX dependencies; import composed stages and export a supported deterministic subset.
4. Build an explicit emissive-light table, environment importance sampling, alias/CDF selection, and unbiased next-event estimation with MIS.
5. Port ReSTIR DI reservoirs only after the reference light estimator passes PDF, visibility, and energy tests.
6. Add persistent GPU buffers, update/refit/IAS paths, CUDA/OpenGL interop, and memory/occupancy telemetry.

### P2: Advanced reconstruction and indirect-light reuse

1. Compare NRD against OptiX on the same motion/AOV contract and published quality/performance corpus.
2. Add ReSTIR GI/PT only after DI, temporal validation, and OpenPBR material probabilities are stable.
3. Prototype neural radiance caching behind an experiment flag; promote only with reproducible dynamic-scene gains and bounded failure behavior.
4. Add capability-gated SER/newer-GPU paths without regressing the RTX 3090 baseline.

## Acceptance Gates

- **Compatibility:** all existing YAML scenes and the full repository test suite continue to pass through explicit legacy translators.
- **USD:** curated `.usda` fixtures preserve prim paths, units, transforms, topology, primvars, instances, cameras, lights, and resolved material bindings through import/export.
- **OpenPBR:** default, dielectric, conductor, roughness, transmission, coat, fuzz, thin-film, emission, and opacity sphere grids are compared with the MaterialX/OpenPBR reference; unsupported active inputs fail explicitly.
- **Physical:** furnace/energy bounds, reciprocity where applicable, normalized PDFs, finite radiance, light-selection PDFs, and CPU/GPU estimator agreement are automated.
- **Temporal:** static convergence, animated lights, camera motion, disocclusion, history reset, and ghosting metrics are automated.
- **Image:** deterministic reference renders record linear EXR values and perceptual error; tone mapping is tested separately from radiance.
- **Performance:** warmup, fixed seeds, clocks/environment, avg/P50/P95/P99, GPU memory, critical path, and work totals are recorded. A speed claim requires equal or better quality at the same scene, resolution, cameras, and sample budget.

## Explicit Non-Goals for the First Implementation Slice

- No handwritten replacement for OpenUSD composition or MaterialX graph parsing.
- No immediate Hydra render delegate; direct stage compilation is the smaller integration boundary.
- No claim of full OpenPBR support while active lobes are ignored.
- No ReSTIR PT or neural cache before the baseline estimator and temporal validation are trustworthy.
- No RTX 40/50-only optimization in the RTX 3090 acceptance path.
